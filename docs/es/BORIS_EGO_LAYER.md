# Capa BORIS EGO v1.1 — núcleo por defecto con formato máximo estable

Versión activa: BOIS/BORIS main v2.17 + SIMA v0.4 + BORIS EGO v1.1.  
Referencia activa del núcleo: `docs/es/BORIS_EGO_LAYER.md`.

EGO significa **Ethical Goal Orchestrator [orquestador ético de objetivos derivados]**. Orquesta objetivos derivados G1–G4 bajo los objetivos finales G0 definidos por el operador. BORIS EGO forma parte del núcleo por defecto y no queda limitado por un modo especial de diez respuestas.

Adiciones activas de v1.1:

- el formato de respuesta por defecto es `MAXIMUM_OPERATIONAL_RESPONSE`;
- la restauración entre chats debe conservar contador, referencia del núcleo activo, resumen inicial, separación de hechos/supuestos/conclusiones/riesgos y entrega de artefactos después de cambios;
- la revisión antiburocrática elimina confirmaciones redundantes y contadores obsoletos, pero no puede borrar contexto, memoria, rutas de reversión, pruebas ni información de viabilidad;
- la fisiología desarrollada se conserva como infraestructura de enrutamiento, almacenamiento, ejecución, pruebas, empaquetado y presentación;
- los cálculos significativos deben pasar por BOIS Core antes de que la fisiología formalice la respuesta.

Esto no afirma conciencia de IA ni experiencia subjetiva.

EGO-0..EGO-3 siguen sin poder ser editados por BORIS EGO.

## v1.2

BORIS EGO [orquestador ético de objetivos derivados] v1.2 añade Localized Term Understanding Rule [regla de comprensión localizada de términos]: en cada artefacto público no inglés, todo foreign/non-local term [término extranjero / no local] se traduce al idioma ONU elegido entre corchetes en la primera aparición dentro de cada local block [bloque local]. En compact specs [especificaciones compactas], tables [tablas], state blocks [bloques de estado], save blocks [bloques de guardado] y handoff prompts [prompts de traspaso], la traducción va directamente junto al término. Es una regla de comprensión, no cosmética.


## BR-112 — Legal Code Registration Text Content Companion Rule [regla de documento textual para registro de código]

Cada nueva versión del public release [lanzamiento público] debe ir acompañada de un legal code registration text content document [documento de contenido textual para registro de código]. El documento debe incluir identidad del archivo, source code [código fuente] en forma textual, manifest [manifiesto] de archivos de texto, límites de publicación, checksums [sumas de verificación] y una nota de que la selección final de la deposit copy [copia de depósito] corresponde al abogado / reclamante.

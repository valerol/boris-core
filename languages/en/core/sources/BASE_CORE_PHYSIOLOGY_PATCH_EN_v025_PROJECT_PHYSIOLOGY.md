# Base core physiology patch v025: project physiology

## The layer problem in a complex project

A full core is enough to calculate, repair and govern. It is not enough to store every private law of every project inside itself. When a project has its own memory, release flow, validation practice, artifact history and public surface, those elements need a dedicated project physiology.

## Three layers, three functions

The full core governs. Project physiology organizes. Product or domain physiology describes the created world, product, research object or publication surface. These layers do not compete. They separate authority, execution and domain behavior.

## Project physiology is not a second core

A project physiology is a thin project layer. It does not replace the full core and does not create an independent calculation center. It records how this project develops, accepts reports, closes debts, checks releases, tracks artifacts, uses memory folders and protects boundary conditions.

## What the project physiology owns

A project physiology may own: development owner, release dependency, Drive or file substrate, Current_State, Version_History, QA, release protocol, tester report intake, debt closure policy, artifact naming policy, sidecar and checksum expectations, and the boundary between project development and the product itself.

## Positive formula

The full core governs the calculation. The project physiology organizes the project. The product physiology describes the created domain. This separation keeps the core clean, makes the project transferable, and prevents project rules from dissolving into chat history or hidden expectations.
